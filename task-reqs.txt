Description
Background
In modern web development, performance is not a feature—it's a necessity. Users expect websites to load quickly and respond smoothly. Poor performance can lead to a negative user experience, lower engagement, and reduced search engine rankings. Google's Core Web Vitals (CWV) have become a standard for measuring the health of a site, focusing on loading (Largest Contentful Paint - LCP), interactivity (First Input Delay - FID, or Total Blocking Time - TBT in lab tests), and visual stability (Cumulative Layout Shift - CLS).

Automating performance testing is essential for maintaining high standards in a continuous integration environment. Tools like Puppeteer, a Node library that provides a high-level API to control headless Chrome, and Lighthouse, an open-source tool for auditing web pages, are industry standards. By integrating these tools, developers can programmatically launch a browser, navigate to pages, and run comprehensive audits to catch performance regressions before they reach production.

This task challenges you to build a functional news portal and, more importantly, to create a robust performance testing suite around it. You will define a performance budget—a set of limits for metrics like load times and Lighthouse scores—and then optimize your application to meet these targets. This process mirrors how professional teams ensure their applications remain fast and reliable.

Follow these steps to complete the project. You will first build the application, then implement the performance testing suite.

Step 2: Build the Articles Listing Page

The list should contain at least 20 article cards to simulate a real-world list.
Each card should link to a unique article detail page (/article/:id).

This page should show more detailed information, including a title, author, date, and a substantial amount of text (at least 500 words) to simulate a real article. Step 4: Implement the Search Page
Create a search input on the articles list page. When a user enters a query and clicks the search button, they should be taken to the /search page, which displays the filtered results. The search functionality can be a simple client-side filter of your mock data. The search page must display the original query and the number of results found.

Step 5: Create the Performance Test Suite
In the /tests/performance directory, create the required test scripts. These scripts will use Puppeteer to launch a headless browser and Lighthouse to audit your application's pages.

Here is a template for a basic Lighthouse test script (lighthouse-articles.test.js):

// /tests/performance/lighthouse-articles.test.js
const puppeteer = require('puppeteer');
const lighthouse = require('lighthouse');
const fs = require('fs');

(async () => {
  const browser = await puppeteer.launch({ headless: true, args: ['--no-sandbox'] });
  const page = await browser.newPage();
  const appUrl = 'http://localhost:3000/articles'; // Adjust port if needed

  await page.goto(appUrl, { waitUntil: 'networkidle0' });

  const { lhr } = await lighthouse(appUrl, {
    port: (new URL(browser.wsEndpoint())).port,
    output: 'json',
    logLevel: 'info',
  });

  // Ensure the reports directory exists
  if (!fs.existsSync('performance-reports')) {
    fs.mkdirSync('performance-reports');
  }

  // Save the report
  fs.writeFileSync('performance-reports/articles-lighthouse.json', JSON.stringify(lhr, null, 2));

  console.log(`Lighthouse score for ${appUrl}: ${lhr.categories.performance.score * 100}`);

  await browser.close();
})();
You will need to create similar scripts for the other required tests, ensuring they generate the specified output files and measure all required metrics. Configure your package.json to have a script that runs all performance tests.

// package.json example
{
  "scripts": {
    "start": "your-start-command",
    "test:performance": "node tests/performance/lighthouse-articles.test.js && node tests/performance/lighthouse-detail.test.js"
  }
}

Implementation Guidelines
Component-Based Architecture: Structure your frontend application using components for reusability and maintainability. For example, create a reusable ArticleCard component.
Mock Data: Create a simple JSON file or an in-memory object to serve as the source for your news articles. This avoids the complexity of a backend API and keeps the focus on the frontend and performance testing.
CSS and Styling: Use a modern styling approach (e.g., CSS Modules, Styled Components, or a utility-first framework like Tailwind CSS). Pay attention to how CSS is loaded, as large CSS files can block rendering.
Optimization: Focus on practical optimization techniques. Ensure images are appropriately sized, and consider using modern image formats like WebP. Use your bundler's production mode to enable minification and tree-shaking.
Error Handling: Your performance tests should be robust. Handle potential timeouts or errors gracefully.
These guidelines are suggestions to help you build a high-quality solution. Your final implementation choices are up to you, as long as they meet the core requirements.

FAQ
Q: Do I need a real backend or database?

A: No. Mock data stored in a local JSON file or hardcoded in your application is sufficient. The focus is on frontend performance, not backend architecture.
Q: How do I generate mock data for the articles?

A: You can use online mock data generators or libraries like Faker.js to create realistic-looking article titles, excerpts, and content. Ensure your article content is long enough (500+ words) to simulate a real page.
Q: My Lighthouse scores fluctuate slightly between runs. Is that normal?

A: Yes, minor fluctuations are normal due to variations in network and CPU conditions. Run your tests multiple times to get a stable average. The verification will allow for a small margin of error, but your application should consistently meet the budget specified in lighthouse-performance-score and lighthouse-accessibility-score.
Q: What if I can't meet the exact performance budget numbers?

A: The performance budgets are a key part of the task. You should actively optimize your application to meet them. Document the optimizations you made in your README. This includes techniques like code splitting, asset compression, and lazy loading (image-lazy-loading).
Q: Can I use a different testing tool instead of Puppeteer/Lighthouse?

A: No. This task specifically requires the use of Puppeteer and Lighthouse to meet the requirements for performance testing and reporting (performance-test-structure, performance-reports-structure).
